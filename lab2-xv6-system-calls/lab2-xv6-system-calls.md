# Lab 2: System Calls

[English](./lab2-xv6-system-calls.md) · [中文](./lab2-xv6-system-calls-zh.md)

In the last lab you used system calls to write a few utilities. In this lab you will add a new system call to xv6, which will help you understand how they work and will expose you to some of the internals of the xv6 kernel. You will add more system calls in later labs.

Before you start coding, read Chapter 2 of the xv6 book, and Sections 4.3 and 4.4 of Chapter 4, and related source files:

- The user-space "stubs" that route system calls into the kernel are in `user/usys.S`, which is generated by `user/usys.pl` when you run `make`. Declarations are in `user/user.h`.
- The kernel-space code that routes a system call to the kernel function that implements it is in `kernel/syscall.c` and `kernel/syscall.h`.
- Process-related code is in `kernel/proc.h` and `kernel/proc.c`.

To start the lab, switch to the `syscall` branch:

```bash
$ git fetch
$ git checkout syscall
$ make clean
```

## Using GDB (easy)

In many cases, print statements will be sufficient to debug your kernel, but sometimes it is useful to single-step through code or get a stack backtrace. The GDB debugger can help.

To get familiar with GDB, run `make qemu-gdb` and then open GDB in another window (see the GDB material on the guidance page). Once you have two windows open, type the following in the GDB window:

```
(gdb) b syscall
Breakpoint 1 at 0x80002142: file kernel/syscall.c, line 243.
(gdb) c
Continuing.
[Switching to Thread 1.2]

Thread 2 hit Breakpoint 1, syscall () at kernel/syscall.c:243
243     {
(gdb) layout src
(gdb) backtrace
```

The `layout` command splits the window in two, showing where GDB is in the source code. `backtrace` prints a stack backtrace.

Answer the following questions in `answers-syscall.txt`:

1. Looking at the backtrace output, which function called `syscall`?

2. Type `n` a few times to step past `struct proc *p = myproc();`. Once past this statement, type `p /x *p`, which prints the current process's `proc` struct (see `kernel/proc.h`) in hex. What is the value of `p->trapframe->a7` and what does that value represent? (Hint: look at `user/init.c`, the first user program xv6 starts, and its compiled assembly `user/init.asm`.)

3. The processor is running in supervisor mode, and we can print privileged registers such as `sstatus` (see RISC-V privileged instructions for a description):
   ```
   (gdb) p /x $sstatus
   ```
   What was the previous mode that the CPU was in?

4. The xv6 kernel contains consistency checks whose failure causes the kernel to panic. Replace the statement `num = p->trapframe->a7;` with `num = * (int *) 0;` at the beginning of `syscall`, run `make qemu`, and you will see something similar to:
   ```
   xv6 kernel is booting

   hart 2 starting
   hart 1 starting
   scause=0xd sepc=0x80001bfe stval=0x0
   panic: kerneltrap
   ```
   Quit out of QEMU. Search for the `sepc` value in `kernel/kernel.asm`. Write down the assembly instruction the kernel is panicking at. Which register corresponds to the variable `num`?

5. Fire up GDB and set a breakpoint at the faulting `epc`:
   ```
   (gdb) b *0x80001bfe
   Breakpoint 1 at 0x80001bfe: file kernel/syscall.c, line 138.
   (gdb) layout asm
   (gdb) c
   ```
   Confirm that the faulting assembly instruction is the same as the one you found above. Why does the kernel crash? (Hint: look at figure 3-3 in the text; is address 0 mapped in the kernel address space? Is that confirmed by the value in `scause`?)

6. To find out which user process was running when the kernel panicked, print the process's name:
   ```
   (gdb) p p->name
   ```
   What is the name of the process that was running when the kernel panicked? What is its process id (`pid`)?

## Sandbox a Command (moderate)

In this assignment you will "sandbox" a process to restrict the system calls it can make. You will create a new `interpose` system call that specifies which system calls the kernel should reject from the calling process.

`interpose` takes two arguments: an integer `mask` and a `path`. The mask's bits specify which system calls to reject. The second argument is used in the next assignment; in this assignment it is always `"-"`. For example, to prevent a process from using the `open` system call:

```c
interpose(1 << SYS_open, "-")
```

The mask should be inherited by children via `fork`.

A user program `user/sandbox.c` is provided that forks, calls `interpose()` in the child, and then execs a program in the child. When correctly implemented:

```
$ sandbox 32768 - cat README
cat: cannot open README
$
```

Your solution is complete when the following passes:

```
$ ./grade-lab-syscall sandbox_mask
== Test sandbox_mask == sandbox_mask: OK (1.5s)
```

### Hints

- Add `$U/_sandbox` to `UPROGS` in `Makefile`.
- Add a prototype for `interpose` to `user/user.h`, a stub to `user/usys.pl`, and a syscall number to `kernel/syscall.h`.
- Add a `sys_interpose()` function in `kernel/sysproc.c` that records the mask in a new field in the `proc` structure (see `kernel/proc.h`). Add `sys_interpose` to the `syscalls` array in `kernel/syscall.c`.
- Modify `kfork()` in `kernel/proc.c` to copy the mask from parent to child.
- Modify `syscall()` in `kernel/syscall.c` to check whether the system call must be rejected.

## Sandbox with Allowed Pathnames (easy)

Extend the sandbox to allow masked `open` and `exec` system calls based on the pathname they use. The second argument of `sys_interpose` is the allowed pathname. If `open` or `exec` is masked but the pathname matches the allowed pathname, those system calls should be permitted.

```
$ sandbox 32768 README grep xv6 README
xv6 is a re-implementation of Dennis Ritchie's and Ken Thompson's Unix
...
$ sandbox 32768 README grep xv6 x
grep: cannot open x
```

## Attack xv6 (moderate)

The xv6 kernel isolates user programs from each other and from the kernel. However, a bug has been intentionally introduced: the `memset(mem, 0, sz)` call in `uvmalloc()` in `kernel/vm.c` is omitted when compiling this lab. Similarly, the two `memset` calls that put garbage into free pages in `kernel/kalloc.c` are omitted. The net effect is that newly allocated memory retains contents from its previous use.

`user/secret.c` writes a secret string into memory and then exits, freeing that memory. Your goal is to add code to `user/attack.c` to find and print the secret that a previous execution of `secret.c` wrote.

Your `attack.c` must work with unmodified xv6 and unmodified `secret.c`.

```
$ secret xyzzy
$ attack
xyzzy
$
```

> The grader runs `attack` twice, and is satisfied if either run produces the secret. The secret strings used in testing contain only digits and upper and lower case letters.

## Difficulty Ratings

Each assignment is marked with one of the following difficulty levels:

**Easy**: Less than an hour. These exercises are typically warm-ups for subsequent exercises.

**Moderate**: 1-2 hours.

**Hard**: More than 2 hours. Often these exercises don't require much code, but the code is tricky to get right.

These times are rough estimates. For some optional assignments we don't have a solution and the difficulty rating is a rough guess. If you find yourself spending more time than expected, please reach out on Piazza or come to office hours.

The exercises generally require only tens to a few hundred lines of code, but the code is conceptually complex and details matter a great deal. Make sure you do the assigned reading for each lab, read through the relevant files, and consult the documentation (RISC-V manuals and other references are on the reference page) before writing any code. Implement your solution in small steps — the assignments often suggest how to break the problem down — and verify each step before proceeding to the next.

> **Warning**: Do not start a lab the night before it is due. It is more time-efficient to spread the work over multiple days. Bugs in an operating system kernel can manifest in bewildering ways and may require substantial thought and careful debugging to understand and fix.


## Debugging Tips

### Understand C and Pointers

The book *The C Programming Language (Second Edition)* by Kernighan and Ritchie is a succinct reference for C. A few pointer idioms are particularly worth remembering:

- If `int *p = (int*)100`, then `(int)p + 1` and `(int)(p + 1)` are different: the first is `101`, the second is `104`. When adding an integer to a pointer, the integer is implicitly multiplied by the size of the object the pointer points to.
- `p[i]` is defined to be the same as `*(p+i)`, referring to the i-th object in memory pointed to by `p`.
- `&p[i]` is the same as `(p+i)`, yielding the address of the i-th object in memory pointed to by `p`.

Whenever you see an addition involving a memory address, ask yourself whether it is integer addition or pointer addition, and make sure the value being added is multiplied appropriately or not.

### Checkpoint Your Progress with Git

If you have an exercise partially working, commit your code. If you break something later, you can roll back to your checkpoint and proceed in smaller steps. Refer to the Git user's manual or a CS-oriented overview of Git to learn more.

### Use Print Statements

If your code fails a test, make sure you understand why. Insert print statements until you understand what is happening.

If your print statements produce a large volume of output, run `make qemu` inside `script` (see `man script`), which logs all console output to a file that you can then search. Don't forget to `exit script` when done.

### Use GDB

Print statements are often sufficient, but sometimes single-stepping through assembly code or inspecting stack variables is necessary. To use GDB with xv6:

1. Run `make qemu-gdb` in one terminal window.
2. Run `gdb-multiarch` (or `riscv64-linux-gnu-gdb` or `riscv64-unknown-elf-gdb`) in another window.
3. Set a breakpoint, then type `c` (continue). xv6 will run until it hits the breakpoint.

> If GDB shows a warning of the form `warning: File ".../.gdbinit" auto-loading has been declined`, edit `~/.gdbinit` to add `add-auto-load-safe-path ...` as suggested by the warning.

### Inspect the Kernel Assembly

To see the assembly code the compiler generates for the xv6 kernel, refer to `kernel/kernel.asm`, which the Makefile produces when compiling the kernel. The Makefile also produces `.asm` files for all user programs.

### Handle Unexpected Kernel Faults

If the kernel causes an unexpected fault, it will print an error message including the program counter (`sepc`) at the point of the crash. You can search `kernel/kernel.asm` to find the function at that address, or run:

```bash
addr2line -e kernel/kernel <pc-value>
```

For a backtrace, restart using GDB: run `make qemu-gdb` in one window, run GDB in another, set a breakpoint at `panic` with `b panic`, then `c` to continue. When the kernel hits the breakpoint, type `bt` to get a backtrace.

### Diagnose Kernel Hangs

If the kernel hangs, possibly due to a deadlock, run `make qemu-gdb` in one window and GDB in another, then `c` to continue. When the kernel appears to hang, press `Ctrl-C` in the `qemu-gdb` window and type `bt` to get a backtrace.

### Use the QEMU Monitor

QEMU provides a monitor for querying the state of the emulated machine. Access it by pressing `Ctrl-A C`. A particularly useful command is `info mem`, which prints the page table. Use the `cpu` command to select which core `info mem` examines, or start QEMU with a single core:

```bash
make CPUS=1 qemu
```
